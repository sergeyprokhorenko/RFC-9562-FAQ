# RFC 9562 FAQ

If for some reason you think UUIDv7 isn't for you, you just don’t know how to use it.


### Appendix C. Informative FAQ

| Topic | Question | Answer |
| :--- | :--- | :--- |
|  | Will UUIDv7 values be strictly increasing (monotonic) when generated in parallel across multiple processes? | Existing implementations may guarantee monotonicity across different processes (with or without limiting throughput), only within a single process or backend (where a sub‑millisecond timestamp segment can improve monotonicity), or not at all. |
|  | What happens if the system clock goes backwards during UUIDv7 generation? | Some implementations handle a clock rollback gracefully by ignoring the rollback and reusing the previous timestamp value until the system clock catches up. Here, monotonicity is provided by a counter and/or by incrementing the timestamp itself as a counter. |
|  | BIGINT or UUIDv7: which one should I choose as the default for my primary keys in a new project? | The default choice for a primary key in a new project should be UUIDv7.<br><br>UUIDv7 delivers search and write performance equivalent to BIGINT while enabling globally unique IDs that can be generated on any client without coordination. UUIDv7 overcomes the shortcomings of auto-increment: difficulty of merging data, key collisions, revealing the total number of records, and making brute‑forcing valid keys easier.<br><br>BIGINT may be chosen only when you are absolutely certain the system will remain a single, monolithic database with very limited storage, and when the primary key will never be exposed publicly. These assumptions are often optimistic and may prove incorrect as the project evolves.<br><br>Migrating from BIGINT to UUIDv7 later is costly and disruptive, whereas starting with UUIDv7 imposes minimal upfront cost. |
|  | Isn't the 16-byte size of UUIDv7 going to bloat my indexes and foreign keys compared to the 8-byte BIGINT? | Yes, UUIDv7 uses exactly twice the storage of BIGINT (16 vs 8 bytes) for the column itself and for every index and foreign key that references it. However, the overall database size does not double because other fields (which may be much larger) remain unchanged, and each row already has a fixed overhead (tuple header) that is identical for both key types.<br><br>More importantly, benchmarks show that UUIDv7 delivers insert and search performance nearly equivalent to BIGINT, thanks to its time‑ordered nature and excellent index locality.<br><br>For the majority of applications, the storage difference is acceptable given the benefits of global uniqueness and client‑side generation. Benchmark‑driven decisions are recommended over theoretical concerns. |
|  | If the system clock jumps far into the future, will all subsequent UUIDv7 values become incorrect and unusable? | Some implementations can handle a forward clock jump by simply incrementing the new, larger timestamp value for the next UUIDs until the system clock catches up. This preserves monotonicity while creating a gap in the sequence. But if the forward clock jump was very large, then when the UUIDv7 generator is restarted, subsequent identifiers will be generated with the correct timestamp. |
|  | Is measuring the raw generation rate of UUIDv7 a useful performance metric? | No. Generating a UUIDv7 is extremely fast compared to the time the database needs to insert or update a row. The generation step is almost never the real bottleneck. |
|  | When many concurrent writers insert rows with UUIDv7 into a database, does the monotonic nature of UUIDv7 cause all new rows to land on the same page, resulting in high page‑level lock contention and slower write throughput? | To avoid it, shift the timestamp of each UUIDv7 by a small offset, assign different offsets to different writers, keep the number of distinct offsets low, and avoid very small offsets. |
|  | Does the time‑ordered nature of UUIDv7 create a hot shard, where all new writes go to the same node, making it unsuitable for some sharded databases? | To eliminate this drawback, derive the sharding key from the UUID as follows: take the integer value of its 32 least significant bits (which are guaranteed random), then divide that value by the number of shards. The remainder determines the target shard. This method works for UUID versions 3, 4, 5, and 7. |
|  | Is the collision probability for UUIDv7 much higher than for UUIDv4 because it uses a timestamp plus only 74–62 random bits? | No. The millisecond timestamp, the monotonic counter, the sub‑millisecond timestamp segment (when present), and the fact that generation typically happens on a single node all drastically reduce the chance of collisions. Under many practical conditions, the collision probability is zero even without considering the random bits. Any calculation of collision risk for UUIDv7 is misleading - the design guarantees uniqueness through its monotonic counter and time components, not through random bits alone. |
|  | Is it a security risk to expose a UUIDv7 primary key in a public API because its timestamp leaks record creation time? | This is a valid concern, but RFC 9562 allows for a solution. By using a timestamp offset that changes randomly over time (similar to FHSS), you can distort the embedded timestamp, thus hiding the true creation time. |
|  | Will generating millions of UUIDv7s per second quickly overflow the monotonic counter and cause collisions? | The standard allows incrementing the timestamp ahead of the actual time to continue generating monotonic values. |
|  | Is partitioning a table on a UUIDv7 column less efficient than using a separate `created_at` column? | No, partitioning directly on the UUIDv7 column is not less efficient, since the most significant bits of UUIDv7 contain the timestamp, and so time‑based partitioning works correctly and efficiently using the id column alone.<br><br>However, in practice a separate `created_at` column is often still needed - not just for partitioning, but because extracting the timestamp from a UUIDv7 for queries is cumbersome and less performant than querying a plain timestamp column. Additionally, the offset of the timestamp in UUIDv7 may be used, which is another reason to keep a dedicated timestamp column. Many applications keep both: the UUIDv7 as the primary key and a separate `created_at` column for easy filtering, reporting, and indexing. The choice to omit `created_at` is possible but usually not recommended unless storage is extremely tight. |
|  | If the system clock is temporarily unavailable, will my UUIDv7 generator completely fail? | Some implementations handle even a long-term unavailability of the system clock by simply incrementing the last available timestamp value until the system clock is restored. |
|  | Is UUIDv7 a fixed 128-bit standard that cannot be extended with my own metadata or checksum without breaking compatibility? | Yes, UUIDv7 is a fixed 128‑bit standard, and you cannot embed your own metadata or checksum directly inside it without breaking compatibility with the specification. Metadata is deliberately excluded from the UUID because metadata can change, while a UUID must remain absolutely stable. Therefore, any associated metadata should accompany the UUID in separate storage locations, such as distinct data fields, external constants, or accompanying parameters. Checksums are not part of the standard since UUIDs are primarily designed for automated processing. |
|  | Will even a minor violation of monotonicity (e.g., a single ID out of order) severely damage my database’s index performance? | No. Minor monotonicity violations have no noticeable impact on database index performance, because UUIDv7's timestamp prefix clusters nearby IDs on the same or adjacent B‑tree pages.<br><br>However, such violations can complicate debugging, cursor‑based pagination, log searching, and any process that relies on strict chronological ordering of IDs. |
|  | Do time-ordered UUIDs have the same negative impact on database `VACUUM` and index maintenance as random UUIDs? | No, this is incorrect. Time-ordered UUIDs (v7) drastically reduce index fragmentation and page splits compared to random UUIDs (v4). By inserting new rows at the right edge of the B-tree, they lead to fewer dead tuples and more efficient `VACUUM` operations, as demonstrated by benchmarks. |
|  | Is pagination with UUIDv7 no different from pagination with any other random or sequential key? | No. UUIDv7 enables highly efficient keyset pagination (cursor-based pagination). Because the IDs are time-ordered, you can paginate by the ID itself using filters like `WHERE id > last_seen_id ORDER BY id`, which is significantly faster than using `OFFSET` and `LIMIT` on a random key. |
| Recovering approximate creation time without a separate `created_at` column | Do you always need a separate `created_at` timestamp column because you cannot get the time from a UUID? | No, that is false. A key feature of UUIDv7 is that the creation time can be extracted directly from the ID using a dedicated function. This allows you to drop the redundant `created_at` column, saving 8 bytes per row and simplifying your schema.<br><br>However, in practice a separate `created_at` column is often still needed, because extracting the timestamp from a UUIDv7 for queries is cumbersome and less performant than querying a plain timestamp column. Additionally, the offset of the timestamp in UUIDv7 may be used, which is another reason to keep a dedicated timestamp column. Many applications keep both: the UUIDv7 as the primary key and a separate `created_at` column for easy filtering, reporting, and indexing. The choice to omit `created_at` is possible but usually not recommended unless storage is extremely tight. |
| Counter overflow | Is a 12-bit monotonic counter for sub-millisecond precision too small and prone to constant overflow? | The 12-bit counter (4,096 values per ms) is generally sufficient for modern systems. However, the RFC allows combining methods, so you can use a longer counter or a combination of a sub-millisecond timestamp fraction and a counter to prevent overflow, even under extremely high generation rates. |
| Feasibility of brute‑force key enumeration when using a long counter | Does the timestamp prefix of a UUIDv7 make it vulnerable to brute-force enumeration attacks? | This is both a risk and a benefit. The time-ordered prefix does make the ID more predictable than a fully random UUIDv4. However, it is far less predictable than a sequential integer. As long as a sufficient number of random bits remain in the ID (at least 32-48), brute-force enumeration is computationally infeasible for most practical purposes. |
| Duplicate objects with different UUIDv7 key values | If two processes generate the same data object, might they create the exact same UUIDv7, leading to a collision? | The probability of a collision is extremely low due to the large amount of random entropy in the UUID. The RFC 9562 specification allows a flexible combination of timestamp, counters, and random bits to ensure monotonicity while maintaining a negligible chance of duplication, even across distributed systems. |
| Comparison during search operations | Is searching and comparing a UUIDv7 column less efficient than searching a BIGINT column? | While comparing two 16-byte UUIDs is inherently more expensive than comparing two 8-byte BIGINTs, the difference in CPU cost is often negligible in real-world I/O-bound database workloads. The far more significant factor is index locality. A UUIDv7 index has far better locality than a random UUIDv4 index, making search operations significantly faster in practice. |
| Protection against timestamp overflow when applying a timestamp offset | Will adding a large offset to a UUIDv7 timestamp overflow the 48-bit field and cause errors? | Yes, this is a real implementation pitfall. For example, in PostgreSQL, an overflow was discovered in the `uuidv7_interval()` function when shifting timestamps for dates far in the future (e.g., beyond the year 2262). Such issues must be handled at the implementation level to ensure safe offset operations. |
| Handling leap seconds in timestamp generation | Will leap seconds cause incorrect timestamps in UUIDv7 and break time ordering? | No. The UUIDv7 specification explicitly excludes leap seconds. It uses a Unix Epoch timestamp measured in milliseconds since midnight, 1 Jan 1970 UTC, leap seconds excluded. This ensures consistent and monotonic time calculation without the irregularities caused by leap seconds. |
| Benchmarking requirements | Are standard benchmarks like a simple `pgbench` test sufficient to evaluate UUIDv7 for my system? | Sergey Prokhorenko argues that standard benchmarks are often low-quality and misleading. He advocates for complex, real-world benchmarks that measure multi-threaded insert rates, query join performance, memory usage, and CPU load under realistic conditions, not just the generation rate of the IDs themselves. |
| Migration from UUIDv4 to UUIDv7 | Is migrating from UUIDv4 to UUIDv7 a risky and difficult process that will break my application? | No, the migration is often surprisingly low-risk. Since UUIDv4 and UUIDv7 are both 128-bit UUIDs, they are indistinguishable to most applications and databases at the storage level. You can often start generating UUIDv7 for new records while keeping existing UUIDv4 IDs, as they are fully compatible and can coexist in the same column. |

|  | Which technique can improve the monotonicity of UUIDv7 generated on remote clients? | The monotonicity of UUIDv7 generated on remote clients can be significantly improved by using a positive timestamp offset equal to half the ping (RTT/2). This offset compensates for the one‑way network delay, making client‑generated UUIDs' timestamps align with the timestamps a server would have assigned upon reception. Server-side generation ensures better monotonicity, though it may need extra client-generated identifiers for integrity and feedback. |
|  | Is it necessary to synchronize the system clocks of UUIDv7 generators with each other? | No, UUIDv7 generators do not need to synchronize their clocks with each other directly. Instead, each generator independently synchronizes its system clock to a global time reference (typically UTC) using standard time synchronization technologies, such as NTP (Network Time Protocol), PTP (Precision Time Protocol), or GPS-based time receivers. This ensures that all generators produce monotonically increasing timestamps without requiring peer‑to‑peer synchronization. |
