# RFC 9562 FAQ


### Appendix C. Informative FAQ

| Original topic (exact wording) | Question / Misconception | Answer |
| :--- | :--- | :--- |
| **Monotonicity violation when generating UUIDv7 in parallel across multiple processes** | “If I generate UUIDv7 in parallel across different processes, the IDs will still be perfectly and globally ordered.” | This is a common misconception. The RFC does not guarantee monotonicity across different processes. Monotonicity is typically guaranteed only within a single process or backend. UUIDs generated in parallel workers can lose sortability and might not be ordered, especially if they have different starting states. |
| **System clock rollback** | “If the system clock goes backwards, my UUIDv7 generator will be stuck, produce invalid IDs, or violate monotonicity.” | Most robust implementations handle a clock rollback gracefully. A common approach is to ignore the rollback and reuse the previous timestamp value until the system clock catches up. However, if the rollback is significant (e.g., >10 seconds), the generator may need to handle it as a critical error. |
| **Choosing between UUIDv7 and BIGINT** | “The main debate is between UUIDv4 and UUIDv7, but the real choice for a primary key should be between UUIDv7 and an integer key like BIGINT.” | Sergey Prokhorenko often states that the comparison with UUIDv4 is not very interesting, as the choice is usually between UUIDv7 and an integer key. BIGINTs (8 bytes) are smaller and faster to generate, but UUIDv7 (16 bytes) offers global uniqueness, distributed generation, and time-ordering. |
| **Disk space consumption** | “A UUID is only 16 bytes, so its disk space consumption is negligible.” | While 16 bytes is small, it is twice the size of a BIGINT (8 bytes) and four times the size of an INTEGER (4 bytes). As a primary key, this size multiplies across all foreign keys and indexes, leading to significantly higher disk and memory I/O, which is a key consideration in the UUIDv7 vs. BIGINT debate. |
| **System clock jumps forward by a large amount** | “If the system clock jumps far into the future, all my subsequent UUIDv7 values will be incorrect and unusable.” | Implementations can handle a forward clock jump by simply using the new, larger timestamp value for the next UUID. This preserves monotonicity but will create a gap in the sequence. Alternatively, the timestamp offset feature can be used to artificially shift the timestamp forward to obscure the real time or to test future scenarios. |
| **Generation latency** | “UUIDv7 generation is so fast that generation latency is never a concern.” | While UUIDv7 generation is fast, latency can be a concern in ultra-low-latency systems, especially when compared to simple integer increments. However, Sergey Prokhorenko has noted that the generation rate is usually excessive and not the bottleneck; the real performance test is the rate of inserting records into a database and joining tables. |
| **Lock contention during parallel UUID generation in multiple processes** | “Generating UUIDv7 in parallel will cause high lock contention, slowing down generation.” | This depends on the implementation. However, an advantage of UUIDv7 over a centralized sequence is that it is designed to be generated locally. Some implementations specifically highlight that UUIDv7 prevents lock contention when generating UUIDs in parallel across multiple processes, as each process can generate IDs independently without a shared lock. |
| **Hotspots during sharding (hot shards)** | “The time-ordered nature of UUIDv7 will create a ‘hot shard’ as all new writes go to the same node, making it unsuitable for sharded databases.” | This is a known trade-off. The same time-ordered prefix that benefits B-tree indexes in a single database can create a write hotspot in a sharded/distributed database. In such architectures, a random or hashed shard key is often preferred to distribute writes evenly across nodes. |
| **Key collision probability** | “Because UUIDv7 uses a timestamp plus only 74–62 random bits, the chance of a collision is much higher than with UUIDv4.” | The probability is still astronomically low for most practical scales. Even with 62 random bits (for high‑frequency generation), you would need to generate billions of IDs per second for centuries to reach a 50% collision risk. The timestamp prefix actually helps avoid collisions across time, and the standard includes counters to make collisions virtually impossible within the same millisecond. |
| **Unwanted disclosure of record creation timestamp** | “Because a UUIDv7 contains a timestamp, using it as a primary key in a public API will leak the exact creation time of every record.” | This is a valid concern, but RFC 9562 allows for a solution. By using a timestamp offset (e.g., the `uuidv7_interval` function), you can distort the embedded timestamp, making it reveal only an approximate time or a future one, thus hiding the true creation time. |
| **Excessively high generation frequency** | “Generating millions of UUIDv7s per second will quickly overflow the monotonic counter and cause collisions.” | The standard provides countermeasures. To handle high-frequency generation, you can implement a “Fixed Bit-Length Dedicated Counter” as per RFC 9562. When the counter overflows within a timestamp, the standard allows incrementing the timestamp ahead of the actual time to continue generating monotonic values. |
| **Table partitioning by a UUIDv7‑formatted key** | “I can partition my table on a UUIDv7 column just like any other column, but it’s less efficient than using a separate `timestamptz` column.” | On the contrary, using the UUIDv7 column as a partition key is highly efficient and a major use case. Since the most significant bits contain the timestamp, you can perform time-based partitioning directly on the `id` column without needing a separate `created_at` timestamp column, saving space and simplifying the schema. |
| **Difference between client‑side generation time and database write time** | “The timestamp in a UUIDv7 is not reliable because it’s generated on the client, which might be before the actual database commit.” | This is correct. The embedded timestamp reflects the time of ID generation on the client, not the transaction commit time on the database server. For strict database-level ordering, server-side generation is more reliable, ensuring better monotonicity. |
| **Temporary unavailability of the system clock** | “If the system clock is unavailable, my UUIDv7 generator will completely fail.” | While a system clock is required for the timestamp, the design is fault-tolerant. A robust implementation can handle a temporary clock failure by using a fallback mechanism, such as a logical counter, to continue generating monotonic IDs until the clock is restored. |
| **Need to add metadata and a checksum** | “UUIDv7 is a fixed 128-bit standard; I cannot extend it with my own metadata or checksum without breaking compatibility.” | While UUIDv7 is 128 bits, the specification is flexible. In practice, implementations can extend UUID-containing identifiers beyond 128 bits to include custom metadata and a checksum if needed for specific applications. |
| **Impact of minor monotonicity violations on database performance** | “Even a minor violation of monotonicity (e.g., a single ID out of order) will severely damage my database’s index performance.” | Minor, infrequent violations of monotonicity generally have no noticeable impact on database performance. The benefits of a mostly time-ordered index are still realized. The primary impact of such violations is on debugging, log searching, and any process that relies on strict chronological ordering of IDs. |
| **Database vacuum impact with time-ordered UUIDs vs. random UUIDs** | “Time-ordered UUIDs have the same negative impact on database `VACUUM` and index maintenance as random UUIDs.” | This is incorrect. Time-ordered UUIDs (v7) drastically reduce index fragmentation and page splits compared to random UUIDs (v4). By inserting new rows at the “right edge” of the B-tree, they lead to fewer dead tuples and more efficient `VACUUM` operations, as demonstrated by benchmarks showing up to 80× less I/O for `VACUUM`. |
| **Pagination by UUIDv7** | “Pagination with UUIDv7 is no different from pagination with any other random or sequential key.” | UUIDv7 enables highly efficient *keyset pagination* (cursor-based pagination). Because the IDs are time-ordered, you can paginate by the ID itself using filters like `WHERE id > last_seen_id ORDER BY id`, which is significantly faster than using `OFFSET` and `LIMIT` on a random key. |
| **Recovering approximate creation time without a separate `created_at` column** | “You always need a separate `created_at` timestamp column because you can’t get the time from a UUID.” | This is false. A key feature of UUIDv7 is that the creation time can be extracted directly from the ID using a dedicated function (e.g., `uuid_extract_time` in PostgreSQL). This allows you to drop the redundant `created_at` column, saving 8 bytes per row and simplifying your schema. |
| **Counter overflow** | “A 12-bit monotonic counter for sub-millisecond precision is too small and will overflow constantly.” | The 12-bit counter (4,096 values per ms) is generally sufficient for modern systems. However, the RFC allows combining methods, so you can use a longer counter or a combination of a sub-millisecond timestamp fraction and a counter to prevent overflow, even under extremely high generation rates. |
| **Feasibility of brute‑force key enumeration when using a long counter** | “The timestamp prefix of a UUIDv7 makes it vulnerable to brute-force enumeration attacks.” | This is both a risk and a benefit. The time-ordered prefix does make the ID more predictable than a fully random UUIDv4. However, it is far less predictable than a sequential integer. As long as a sufficient number of random bits remain in the ID (at least 32-48), brute-force enumeration is computationally infeasible for most practical purposes. |
| **Duplicate objects with different UUIDv7 key values** | “If two processes generate the same data object, they might create the exact same UUIDv7, leading to a collision.” | The probability of a collision is extremely low due to the large amount of random entropy in the UUID. The RFC 9562 specification allows a flexible combination of timestamp, counters, and random bits to ensure monotonicity while maintaining a negligible chance of duplication, even across distributed systems. |
| **Comparison during search operations** | “Searching and comparing a UUIDv7 column is less efficient than searching a BIGINT column.” | While comparing two 16-byte UUIDs is inherently more expensive than comparing two 8-byte BIGINTs, the difference in CPU cost is often negligible in real-world I/O-bound database workloads. The far more significant factor is index locality. A UUIDv7 index has far better locality than a random UUIDv4 index, making search operations significantly faster in practice. |
| **Protection against timestamp overflow when applying a timestamp offset** | “Adding a large offset to a UUIDv7 timestamp will overflow the 48-bit field and cause errors.” | This is a real implementation pitfall. For example, in PostgreSQL, an overflow was discovered in the `uuidv7_interval()` function when shifting timestamps for dates far in the future (e.g., beyond the year 2262). Such issues must be handled at the implementation level to ensure safe offset operations. |
| **Handling leap seconds in timestamp generation** | “Leap seconds will cause incorrect timestamps in UUIDv7, breaking time ordering.” | The UUIDv7 specification explicitly excludes leap seconds. It uses a Unix Epoch timestamp measured in milliseconds since midnight, 1 Jan 1970 UTC, **leap seconds excluded**. This ensures consistent and monotonic time calculation without the irregularities caused by leap seconds. |
| **Benchmarking requirements** | “Standard benchmarks like a simple `pgbench` test are sufficient to evaluate UUIDv7 for my system.” | Sergey Prokhorenko argues that standard benchmarks are often low-quality and misleading. He advocates for complex, real-world benchmarks that measure multi-threaded insert rates, query join performance, memory usage, and CPU load under realistic conditions, not just the generation rate of the IDs themselves. |
| **Migration from UUIDv4 to UUIDv7** | “Migrating from UUIDv4 to UUIDv7 is a risky and difficult process that will break my application.” | The migration is often surprisingly low-risk. Since UUIDv4 and UUIDv7 are both 128-bit UUIDs, they are indistinguishable to most applications and databases at the storage level. You can often start generating UUIDv7 for new records while keeping existing UUIDv4 IDs, as they are fully compatible and can coexist in the same column. |

---

**Note on sorting:** The order above reflects the approximate frequency and intensity of real-world discussions (based on public PostgreSQL lists, Stack Overflow, GitHub issues, and Sergey Prokhorenko’s commentary). Topics such as monotonicity, clock rollback, vs. BIGINT, disk space, and collisions tend to appear most often.
