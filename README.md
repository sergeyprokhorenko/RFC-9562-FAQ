# RFC 9562 FAQ

RFC 9562 FAQ: Common concerns about UUIDv7, answered. If you think UUIDv7 doesn’t fit your use case, this guide might show you a new perspective.


## Appendix C. Informative FAQ

### Choosing UUIDv7

| # | Question | Answer |
| :---: | :--- | :--- |
| 1 | BIGINT or UUIDv7: which one should I choose as the default for my primary keys in a new project? | The default choice for a primary key in a new project should be UUIDv7.<br><br>UUIDv7 delivers search and write performance equivalent to BIGINT while enabling globally unique IDs that can be generated on any client without coordination. UUIDv7 overcomes the shortcomings of auto-increment: difficulty of merging or comparing data from different sources, key collisions, revealing the total number of records, and making brute-forcing valid keys easier. UUIDv7 also provides identifier-based full-text search and easier logging, auditing, and debugging. It is the best choice for time-series databases.<br><br>BIGINT may be chosen only when you are absolutely certain the system will remain a single, monolithic database with very limited storage, and when the primary key will never be exposed publicly. These assumptions are often optimistic and may prove incorrect as the project evolves.<br><br>Migrating from BIGINT to UUIDv7 later is costly and disruptive, whereas starting with UUIDv7 imposes minimal upfront cost. |
| 2 | When to choose UUIDv7 vs UUIDv4? | UUIDv7 is best for primary keys in tables that handle a lot of writes. Because it's time-ordered, it improves index performance and reduces page splits in databases like PostgreSQL or MySQL.<br><br>UUIDv4 should only be used when you need purely random identifiers with no sortable or time-based structure, for example, session tokens, API keys, or one-time use codes where ordering doesn't matter. |
| 3 | Is it necessary to implement multiple UUID versions in DBMS and system libraries? | No. As a rule, implementing only UUIDv7 is sufficient for almost all new applications. UUIDv1, UUIDv2, UUIDv3, and UUIDv5 were created in 2005 without a clear understanding of real-world requirements. They exist for variety, not because each solves a distinct problem. UUIDv6 was added solely to extend the life of deployments built on UUIDv1, not as a standalone new version. UUIDv4 is kept only for legacy systems that cannot adopt an advanced implementation of UUIDv7. RFC 9562 retains all versions merely to avoid invalidating existing usage, never to recommend implementing the full set. No regulatory or procurement document requires coverage of every version. RFC 9562 treats UUIDs as opaque values, so a generator that produces all versions adds nothing for parsing or wire compatibility. All effort should go into an advanced UUIDv7 implementation.<br><br>UUIDv1 and UUIDv2 rely on MAC addresses and are considered problematic for security and privacy in modern distributed systems.<br><br>UUIDv3 uses MD5, which is cryptographically broken, so it should never be used in any new system.<br><br>UUIDv4 works well only in the rare cases where ordering is not required. UUIDv7, by contrast, works in all cases and can fully replace UUIDv4, provided a clock is available. A recommended technique supported in PostgreSQL and other systems is to apply random timestamp offsets to UUIDv7 to hide the true creation time while preserving time ordering. The lower 32 random bits of UUIDv7 can also serve as a sharding key.<br><br>UUIDv5 uses SHA-1, which is obsolete and insecure (though less broken than MD5). It lacks time ordering and index locality. Some might argue that deterministic, name-based UUIDv5 is useful, but in practice no entity is truly immutable. Email addresses, names, and even URIs can change over time. With UUIDv7, you store the external identifier (for example, an email address) in a separate column and keep a lookup table. The UUIDv7 itself never needs to change. This is why UUIDv7 can fully replace UUIDv5 for new designs.<br><br>UUIDv6 is unpopular, and its timestamp structure differs from UUIDv1, making it difficult to use UUIDv6 as a replacement for UUIDv1. Therefore UUIDv7 replaces it without any drawback.<br><br>UUIDv8 is custom, therefore it cannot be uniformly implemented across database management systems and system libraries. |

### Storage and Performance

| # | Question | Answer |
| :---: | :--- | :--- |
| 4 | Isn't the 16-byte size of UUIDv7 going to bloat my indexes and foreign keys compared to the 8-byte BIGINT? | Yes, UUIDv7 uses exactly twice the storage of BIGINT (16 vs 8 bytes) for the column itself and for every index and foreign key that references it. However, the overall database size does not double because other fields (which may be much larger) remain unchanged, and each row already has a fixed overhead (tuple header) that is identical for both key types.<br><br>More importantly, benchmarks show that UUIDv7 delivers insert and search performance nearly equivalent to BIGINT, thanks to its time-ordered nature and excellent index locality.<br><br>For the majority of applications, the storage difference is acceptable given the benefits of global uniqueness and client-side generation. Benchmark-driven decisions are recommended over theoretical concerns. |
| 5 | If I use UUIDv7, can I generate IDs on the client-side without a round-trip to the database, unlike a BIGINT? | Yes. UUIDv7 generates the timestamp and random bits locally, so no central sequence or database call is needed. A BIGINT typically requires a database request for the next value. |
| 6 | Does the time-ordered nature of UUIDv7 create a hot shard, where all new writes go to the same node, making it unsuitable for some sharded databases? | Yes, if you use the full UUIDv7 value as a sharding key, the time-ordered prefix can cause all writes to concentrate on a single shard for a given time period. This is because the most significant bits change slowly over time.<br><br>However, this problem is easily avoided: use only the 32 least significant bits (the random part) as the sharding key instead of the entire UUID. This method works for UUID versions 3, 4, 5, and 7. At the same time, choosing a more meaningful sharding key, such as a client ID, is preferable to focus related computations on the same shard. |
| 7 | Does time-ordered UUIDv7 have the same negative impact on database VACUUM and index maintenance as random UUIDv4? | No. UUIDv7 offers significantly better performance for B-tree indexes. Because UUIDv7 values are generated in a sortable, time-ordered sequence, new rows are appended to the end of the index rather than scattered randomly throughout. This behavior, unlike UUIDv4, minimizes page splits, reduces index fragmentation and bloat, and lessens the workload on VACUUM. |
| 8 | When many concurrent writers insert rows with UUIDv7 into a database, does the monotonic nature of UUIDv7 cause all new rows to land on the same page, resulting in high page-level lock contention and slower write throughput? | To avoid it, shift the timestamp of each UUIDv7 by a small offset, assign different offsets to different writers, keep the number of distinct offsets low, and avoid very small offsets. |
| 9 | How do I speed up UUIDv7 generation for high-throughput LLM pipelines? | Use thread-local UUID generators to eliminate locking and atomic operations.<br><br>Pre-allocate randomness. Fill a circular buffer from your CSPRNG and draw bits on each UUID. Refill the buffer from the CSPRNG whenever the read cursor would wrap around, so every byte is used exactly once. This turns costly CSPRNG calls into cheap memory reads.<br><br>Replace fresh randomness per UUID with sub-millisecond timestamp fraction (12-bit of rand_a) and monotonic counter (30 bits of rand_b). Re-seed the counter to a random value at the start of each millisecond, and force its most significant bit to zero. Leave the lower 32 bits for randomness.<br><br>If the 30‑bit counter overflows (extreme throughput), increment the combined 60‑bit timestamp (48‑bit ms + 12‑bit sub‑ms fraction) by 1 and reset the counter. This minimal forward jump keeps the UUIDs monotonically increasing while staying as close to wall‑clock time as possible.<br><br>Obtain the 48-bit Unix millisecond timestamp lazily: check on each UUID whether the millisecond has changed (for example, by using rdtsc/rdtscp and a known ticks-per-ms ratio, or a fast monotonic clock). Only when a new millisecond is detected, refresh the cached timestamp. Use rdtsc for the 12-bit sub-millisecond fraction. This eliminates all per-UUID system-clock calls.<br><br>Swap the CSPRNG for a fast non-crypto PRNG. Not recommended for production, as this weakens uniqueness and nonpredictability guarantees. |
| 10 | Will UUIDv7 values be strictly increasing (monotonic) when generated in parallel across multiple processes? | It depends on the implementation. Existing implementations may guarantee monotonicity across different processes (with or without limiting throughput), only within a single process or backend (where a sub-millisecond timestamp segment can improve monotonicity), or not at all. |

### Security and Privacy

| # | Question | Answer |
| :---: | :--- | :--- |
| 11 | Is it a security risk to expose a UUIDv7 primary key in a public API because its timestamp leaks record creation time? | This is a valid concern, but implementations are allowed to use a timestamp offset that changes randomly over time (similar to FHSS) that distorts the embedded timestamp, thus hiding the true creation time. |
| 12 | Is UUIDv7 vulnerable to brute-force enumeration attacks when using a long counter? | No. Even with a long counter (up to 42 bits), UUIDv7 remains resistant to brute-force enumeration. The counter is randomly seeded per timestamp tick, and the ID retains at least 32 bits of dedicated randomness plus the counter’s entropy. Combined with constantly changing timestamps, brute-force search is computationally infeasible for practical purposes. A CSPRNG is recommended. |
| 13 | Some UUIDv7 implementations allow generation from a specific timestamp, while others offer a timestamp offset. Considering they might seem interchangeable, which approach is preferable and why? | Choose timestamp offset, because generation from a specific timestamp is a safety anti-pattern. Unintentional passing of an arbitrary (fixed or decreasing) timestamp to uuidv7() fundamentally breaks monotonicity, the backbone of time-based sortable UUIDs. So the result cannot be considered a valid UUIDv7 implementation. |
| 14 | Is the collision probability for UUIDv7 much higher than for UUIDv4 because it uses a timestamp plus only 74–62 random bits? | No. Some analyses suggest a slightly higher theoretical collision probability for UUIDv7 (74 random bits) than for UUIDv4 (122 random bits). However, for values generated at different timestamps or with different counter values, collision is impossible. Therefore, the practical collision risk remains vanishingly small and comparable to UUIDv4 for all real-world workloads. For a monotonically increasing sequence of UUIDv7 values generated on the same system, collision is impossible. |
| 15 | Should generated UUIDs be checked for collisions or should appropriate constraints be set in the database? | No. With a CSPRNG, the collision probability is so negligibly small that neither explicit collision checks nor a database UNIQUE constraint are needed. |

### Clock and Time Handling

| # | Question | Answer |
| :---: | :--- | :--- |
| 16 | What happens if the system clock goes backwards during UUIDv7 generation? | Some implementations handle a clock rollback gracefully by ignoring the rollback and reusing the previous timestamp value until the system clock catches up. Here, monotonicity is provided by a counter and/or by incrementing the timestamp itself as a counter. |
| 17 | If the system clock jumps far into the future, will all subsequent UUIDv7 values become incorrect and unusable? | Some implementations can handle a forward clock jump by simply incrementing the new, larger timestamp value for subsequent UUIDs until the system clock catches up. This preserves monotonicity while creating a time gap. Applications should not rely on the UUIDv7 timestamp to correspond to the actual date and time of generation. However, when the UUIDv7 generator is restarted, subsequent identifiers will again be generated with the correct timestamp. |
| 18 | If the system clock is temporarily unavailable, will my UUIDv7 generator completely fail? | Some implementations handle even a long-term unavailability of the system clock by simply incrementing the last available timestamp value until the system clock is restored. |
| 19 | Is it necessary to synchronize the system clocks of UUIDv7 generators with each other? | No, UUIDv7 generators do not need to synchronize their clocks with each other directly. Instead, each generator independently synchronizes its system clock to a global time reference (typically UTC) using standard time synchronization technologies, such as NTP (Network Time Protocol), PTP (Precision Time Protocol), or GPS-based time receivers. This ensures that all generators produce monotonically increasing timestamps without requiring peer-to-peer synchronization. |
| 20 | Will leap seconds cause incorrect timestamps in UUIDv7 and break time ordering? | No. UUIDv7 uses Unix time (leap seconds excluded, as mandated by RFC 9562), so the timestamp is strictly monotonic and ordering remains correct. |

### Monotonic Counter and Overflow

| # | Question | Answer |
| :---: | :--- | :--- |
| 21 | How do UUIDv7 implementations preserve monotonic order at very high generation frequencies? | There are three methods that can be combined. The first uses a randomly seeded counter that increments for each UUID generated within the same millisecond. The second optionally uses the 12-bit rand_a as a sub-millisecond fraction, giving approximately 244 ns resolution. The third increments the timestamp itself under extreme generation rates. |
| 22 | Will generating millions of UUIDv7s per second quickly overflow the monotonic counter and cause collisions? | The standard allows incrementing the timestamp ahead of the actual time to continue generating monotonic values. |
| 23 | Is a 12-bit monotonic counter for sub-millisecond precision too small and prone to constant overflow? | That is a fair concern, however implementations increment the timestamp as a counter in extreme situations or extend the counter up to 42 bits. In this case, the most significant bit of the counter is typically initialized to zero every millisecond. Therefore, counter overflows are impossible. |
| 24 | Will adding a large offset to a UUIDv7 timestamp overflow the 48-bit field and cause errors? | In some implementations, the timestamp is treated as a circular buffer. If an offset pushes the timestamp beyond the maximum allowed value, it wraps around to the minimum value plus the excess. Likewise, if a negative offset is applied and its absolute value exceeds the time elapsed since the Unix epoch, the timestamp wraps around to the maximum value minus the deficit (the amount by which the absolute offset exceeds the elapsed time). The function always returns a valid UUID and never raises an overflow error. |

### Schema and Indexing

| # | Question | Answer |
| :---: | :--- | :--- |
| 25 | Do I need a separate created_at column if UUIDv7 already contains a timestamp? | In some practical scenarios, yes. While some database systems offer generated columns that can automatically extract date and time from a UUIDv7, or you may use corresponding functions, the date and time extracted from a UUIDv7 may differ substantially from the real wall-clock time at the moment of generation. RFC 9562 explicitly allows the timestamp used during generation to be shifted by an arbitrary offset (for example, to obscure the actual creation time) or to be affected by system clock issues and timestamp increment. |
| 26 | Is pagination with UUIDv7 no different from pagination with any other random or sequential key? | No. UUIDv7 enables highly efficient keyset pagination (cursor-based pagination). Because the IDs are time-ordered, you can paginate by the ID itself using filters like WHERE id > last_seen_id ORDER BY id, which is significantly faster than using OFFSET and LIMIT on a random key. |
| 27 | Is searching and comparing a UUIDv7 column less efficient than searching a BIGINT column? | It depends on the implementation. When comparing UUIDv7, only a few high-order bits may need to be examined until the first mismatch is found, which greatly reduces the time required for comparison and searching. |
| 28 | Is partitioning a table by a UUIDv7 column less efficient than using a separate created_at column? | No. Partitioning directly by a UUIDv7 column (or by UUIDv7's most significant bits) is efficient and requires no extra storage. The timestamp embedded in the most significant bits of UUIDv7 naturally maintains chronological order, making the column well-suited for time-based range partitions without any auxiliary timestamp column. |
| 29 | Will even a minor violation of monotonicity (e.g., a single ID out of order) severely damage my database’s index performance? | No. Minor monotonicity violations have no noticeable impact on database index performance, because UUIDv7's timestamp prefix clusters nearby IDs on the same or adjacent B-tree pages.<br><br>However, such violations can complicate debugging, cursor-based pagination, log searching, Change Data Capture (CDC) pipelines, and any process that relies on strict chronological ordering of IDs. They can also cause incorrect results in window functions by distorting the perceived event sequence. |
| 30 | How can I improve the monotonicity of UUIDv7 generated on remote clients for specific use cases like reliable cursor-based pagination, if strict monotonicity is critical? | The monotonicity of UUIDv7 generated on remote clients can be significantly improved by using a positive timestamp offset equal to half the current ping time (RTT/2). This offset compensates for the one-way network delay, making the timestamps of client-generated UUIDs align with the timestamps a server would have assigned upon reception. Server-side generation ensures better monotonicity, though it may require additional client-generated identifiers for integrity and feedback. |

### Benchmarks

| # | Question | Answer |
| :---: | :--- | :--- |
| 31 | Are standard benchmarks like a simple pgbench test sufficient to evaluate UUIDv7 for my system? | Standard benchmarks like a simple pgbench run are not sufficient to properly evaluate UUIDv7 for your system. A more thorough, real-world assessment is recommended that measures multi-threaded insert throughput with UUIDv7 generation and, importantly, the performance of subsequent queries such as LEFT JOIN and INNER JOIN. These tests should be conducted under realistic conditions that reflect your actual environment, including hardware specifications, row count, and average row size within the table. When using pgbench, it should be configured with a custom script file that simulates these exact workloads, allowing for a meaningful comparison between UUIDv7 and its practical alternatives. |
| 32 | Is measuring the raw generation rate of UUIDv7 a useful performance metric? | Usually, no. Generating a UUIDv7 with a fast entropy source is extremely fast compared to the time the database needs to insert or update a row. The generation step is almost never the real bottleneck. See the answer about standard benchmarks to properly evaluate UUIDv7 performance. |

### Migration to UUIDv7

| # | Question | Answer |
| :---: | :--- | :--- |
| 33 | How to migrate from BIGINT to UUIDv7 with zero downtime? | Add a nullable UUIDv7 column to the main table and a matching nullable UUIDv7 foreign-key column to every child table that references the old BIGINT. For new rows, generate a standard UUIDv7 for the main record and write it into both the main row and the related child rows while still populating the old BIGINT columns as usual. Leave existing rows with NULL in the new columns. Backfill the main table’s old rows with UUIDv7. Use an optional negative timestamp offset only if you want the older IDs to sort before the newer ones. Then backfill child tables by joining on the old BIGINT key to set the new UUIDv7 foreign keys. Create partial indexes ignoring NULLs on all new UUIDv7 columns to keep queries fast during the transition. Once backfill is verified, make the new columns NOT NULL, switch the application and foreign-key constraints to UUIDv7, and finally drop the old BIGINT columns. |
| 34 | Is migration from UUIDv4 to UUIDv7 a risky and difficult process that will break my application? | No, the migration is often surprisingly low-risk and easy. Since UUIDv4 and UUIDv7 are both 128-bit UUIDs, they are indistinguishable to most applications and databases at the storage level. You can typically start generating UUIDv7 for new records while keeping existing UUIDv4 IDs. They are fully compatible and can coexist in the same column, so cascading key updates are unnecessary.<br><br>However, be aware of three caveats:<br><br>If your application extracts the timestamp from UUIDs, existing UUIDv4 values will not contain a valid timestamp.<br><br>You may need to remove any constraints that enforce a specific UUID version.<br><br>If you shard using the full UUID as a sharding key, switching to UUIDv7 can create hotspots. In that case, use only the 32 least significant bits (the random part) as the sharding key. |
| 35 | Does Guid.CreateVersion7() in .NET 9 produce a sortable UUIDv7 for SQL Server uniqueidentifier columns, and if not, what is the correct alternative? | No. Guid.CreateVersion7() stores the timestamp in little-endian order, violating RFC 9562's normal big-endian requirement, and SQL Server's uniqueidentifier type reorders bytes during comparison, both of which prevent lexicographical time-sorting. To achieve correct chronological sorting, generate UUIDv7 using a third-party library that follows the RFC (e.g., UUIDNext, Medo.Uuid7, Costasdev.Uuidv7) and store the 16-byte array in a binary(16) column instead of uniqueidentifier. |
| 36 | Is UUIDv7 a fixed 128-bit standard that cannot be extended with my own metadata or checksum without breaking compatibility? | Yes, UUIDv7 is a fixed 128-bit standard, and you cannot embed your own metadata or checksum directly inside it without breaking compatibility with the specification. Metadata is deliberately excluded from the UUID because metadata can change, while a UUID must remain absolutely stable. Therefore, any associated metadata should accompany the UUID in separate storage locations, such as distinct data fields, external constants, or accompanying parameters. TOON (for LLM) and JSON metadata formats are recommended for use outside databases. Checksums are not part of the standard since UUIDs are primarily designed for automated processing. |
