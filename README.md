# RFC 9562 FAQ

If for some reason you think UUIDv7 isn't for you, you just don’t know how to use it.


### Appendix C. Informative FAQ

| # | Question | Answer |
| :---: | :--- | :--- |
| 1 | Will UUIDv7 values be strictly increasing (monotonic) when generated in parallel across multiple processes? | It depends on the implementation. Existing implementations may guarantee monotonicity across different processes (with or without limiting throughput), only within a single process or backend (where a sub‑millisecond timestamp segment can improve monotonicity), or not at all. |
| 2 | What happens if the system clock goes backwards during UUIDv7 generation? | Some implementations handle a clock rollback gracefully by ignoring the rollback and reusing the previous timestamp value until the system clock catches up. Here, monotonicity is provided by a counter and/or by incrementing the timestamp itself as a counter. |
| 3 | BIGINT or UUIDv7: which one should I choose as the default for my primary keys in a new project? | The default choice for a primary key in a new project should be UUIDv7.<br><br>UUIDv7 delivers search and write performance equivalent to BIGINT while enabling globally unique IDs that can be generated on any client without coordination. UUIDv7 overcomes the shortcomings of auto-increment: difficulty of merging or comparing data from different sources, key collisions, revealing the total number of records, and making brute‑forcing valid keys easier. UUIDv7 also provides identifier-based full-text search and easier logging, auditing, and debugging. It is the best choice for time-series databases.<br><br>BIGINT may be chosen only when you are absolutely certain the system will remain a single, monolithic database with very limited storage, and when the primary key will never be exposed publicly. These assumptions are often optimistic and may prove incorrect as the project evolves.<br><br>Migrating from BIGINT to UUIDv7 later is costly and disruptive, whereas starting with UUIDv7 imposes minimal upfront cost. |
| 4 | Isn't the 16-byte size of UUIDv7 going to bloat my indexes and foreign keys compared to the 8-byte BIGINT? | Yes, UUIDv7 uses exactly twice the storage of BIGINT (16 vs 8 bytes) for the column itself and for every index and foreign key that references it. However, the overall database size does not double because other fields (which may be much larger) remain unchanged, and each row already has a fixed overhead (tuple header) that is identical for both key types.<br><br>More importantly, benchmarks show that UUIDv7 delivers insert and search performance nearly equivalent to BIGINT, thanks to its time‑ordered nature and excellent index locality.<br><br>For the majority of applications, the storage difference is acceptable given the benefits of global uniqueness and client‑side generation. Benchmark‑driven decisions are recommended over theoretical concerns. |
| 5 | If the system clock jumps far into the future, will all subsequent UUIDv7 values become incorrect and unusable? | Some implementations can handle a forward clock jump by simply incrementing the new, larger timestamp value for subsequent UUIDs until the system clock catches up. This preserves monotonicity while creating a time gap. Applications should not rely on the UUIDv7 timestamp to correspond to the actual date and time of generation. However, when the UUIDv7 generator is restarted, subsequent identifiers will again be generated with the correct timestamp. |
| 6 | Is measuring the raw generation rate of UUIDv7 a useful performance metric? | Usually, no. Generating a UUIDv7 with a fast entropy source is extremely fast compared to the time the database needs to insert or update a row. The generation step is almost never the real bottleneck. See the answer about standard benchmarks to properly evaluate UUIDv7 performance. |
| 7 | When many concurrent writers insert rows with UUIDv7 into a database, does the monotonic nature of UUIDv7 cause all new rows to land on the same page, resulting in high page‑level lock contention and slower write throughput? | To avoid it, shift the timestamp of each UUIDv7 by a small offset, assign different offsets to different writers, keep the number of distinct offsets low, and avoid very small offsets. |
| 8 | Does the time‑ordered nature of UUIDv7 create a hot shard, where all new writes go to the same node, making it unsuitable for some sharded databases? | Yes, if you use the full UUIDv7 value as a sharding key, the time-ordered prefix can cause all writes to concentrate on a single shard for a given time period. This is because the most significant bits change slowly over time.<br><br>However, this problem is easily avoided: use only the 32 least significant bits (the random part) as the sharding key instead of the entire UUID. This method works for UUID versions 3, 4, 5, and 7. |
| 9 | Is the collision probability for UUIDv7 much higher than for UUIDv4 because it uses a timestamp plus only 74–62 random bits? | No. Some analyses suggest a slightly higher theoretical collision probability for UUIDv7 (74 random bits) than for UUIDv4 (122 random bits). However, for values generated at different timestamps or with different counter values, collision is impossible. Therefore, the practical collision risk remains vanishingly small and comparable to UUIDv4 for all real-world workloads. For a monotonically increasing sequence of UUIDv7 values generated on the same system, collision is impossible. |
| 10 | Is it a security risk to expose a UUIDv7 primary key in a public API because its timestamp leaks record creation time? | This is a valid concern, but RFC 9562 allows for a solution. By using a timestamp offset that changes randomly over time (similar to FHSS), you can distort the embedded timestamp, thus hiding the true creation time. |
| 11 | Will generating millions of UUIDv7s per second quickly overflow the monotonic counter and cause collisions? | The standard allows incrementing the timestamp ahead of the actual time to continue generating monotonic values. |
| 12 | Do I need a separate `created_at` column if UUIDv7 already contains a timestamp? | In most practical scenarios, yes. While some database systems offer generated columns that can automatically extract date and time from a UUIDv7, or you may use corresponding functions, the date and time extracted from a UUIDv7 may differ substantially from the real wall-clock time at the moment of generation. RFC 9562 explicitly allows the timestamp used during generation to be shifted by an arbitrary offset (for example, to obscure the actual creation time) or to be affected by system clock issues. |
| 13 | If the system clock is temporarily unavailable, will my UUIDv7 generator completely fail? | Some implementations handle even a long-term unavailability of the system clock by simply incrementing the last available timestamp value until the system clock is restored. |
| 14 | Is UUIDv7 a fixed 128-bit standard that cannot be extended with my own metadata or checksum without breaking compatibility? | Yes, UUIDv7 is a fixed 128‑bit standard, and you cannot embed your own metadata or checksum directly inside it without breaking compatibility with the specification. Metadata is deliberately excluded from the UUID because metadata can change, while a UUID must remain absolutely stable. Therefore, any associated metadata should accompany the UUID in separate storage locations, such as distinct data fields, external constants, or accompanying parameters. Checksums are not part of the standard since UUIDs are primarily designed for automated processing. |
| 15 | Will even a minor violation of monotonicity (e.g., a single ID out of order) severely damage my database’s index performance? | No. Minor monotonicity violations have no noticeable impact on database index performance, because UUIDv7's timestamp prefix clusters nearby IDs on the same or adjacent B‑tree pages.<br><br>However, such violations can complicate debugging, cursor‑based pagination, log searching, and any process that relies on strict chronological ordering of IDs. |
| 16 | Do time-ordered UUIDs have the same negative impact on database `VACUUM` and index maintenance as random UUIDs? | No, this is incorrect. Time-ordered UUIDs (v7) drastically reduce index fragmentation and page splits compared to random UUIDs (v4). By inserting new rows at the right edge of the B-tree, they lead to fewer dead tuples and more efficient `VACUUM` operations, as demonstrated by benchmarks. |
| 17 | Is pagination with UUIDv7 no different from pagination with any other random or sequential key? | No. UUIDv7 enables highly efficient keyset pagination (cursor-based pagination). Because the IDs are time-ordered, you can paginate by the ID itself using filters like `WHERE id > last_seen_id ORDER BY id`, which is significantly faster than using `OFFSET` and `LIMIT` on a random key. |
| 18 | Is partitioning a table by a UUIDv7 column less efficient than using a separate created_at column? | No. Partitioning directly by a UUIDv7 column is efficient and requires no extra storage. The timestamp embedded in the most significant bits of UUIDv7 naturally maintains chronological order, making the column well-suited for time-based range partitions without any auxiliary timestamp column. |
| 19 | Is a 12-bit monotonic counter for sub-millisecond precision too small and prone to constant overflow? | That could be true. However, implementations increment the timestamp as a counter in extreme situations or extend the counter up to 42 bits. In this case, the most significant bit of the counter is typically initialized to zero every millisecond. Therefore, counter overflows are impossible. |
| 20 | Is UUIDv7 vulnerable to brute-force enumeration attacks when using a long counter? | No. Even with a long counter (up to 42 bits), UUIDv7 remains resistant to brute‑force enumeration. The counter is randomly seeded per timestamp tick, and the ID retains at least 32 bits of dedicated randomness plus the counter’s entropy. Combined with constantly changing timestamps, brute‑force search is computationally infeasible for practical purposes. |
| 21 | Is searching and comparing a UUIDv7 column less efficient than searching a BIGINT column? | It depends on the implementation. When comparing UUIDv7, only a few high-order bits may need to be examined until the first mismatch is found, which greatly reduces the time required for comparison and searching. |
| 22 | Will adding a large offset to a UUIDv7 timestamp overflow the 48-bit field and cause errors? | In some implementations, the timestamp is treated as a circular buffer. If an offset pushes the timestamp beyond the maximum allowed value, it wraps around to the minimum value plus the excess. Likewise, if a negative offset is applied and its absolute value exceeds the time elapsed since the Unix epoch, the timestamp wraps around to the maximum value minus the deficit (the amount by which the absolute offset exceeds the elapsed time). The function always returns a valid UUID and never raises an overflow error. |
| 23 | Will leap seconds cause incorrect timestamps in UUIDv7 and break time ordering? | No. UUIDv7 uses Unix time (leap seconds excluded, as mandated by RFC 9562), so the timestamp is strictly monotonic and ordering remains correct. |
| 24 | Are standard benchmarks like a simple `pgbench` test sufficient to evaluate UUIDv7 for my system? | Standard benchmarks like a simple `pgbench` run are **not sufficient** to properly evaluate UUIDv7 for your system. A more thorough, real-world assessment is recommended that measures multi-threaded insert throughput with UUIDv7 generation and, importantly, the performance of subsequent queries such as `LEFT JOIN` and `INNER JOIN`. These tests should be conducted under realistic conditions that reflect your actual environment, including hardware specifications, row count, and average row size within the table. When using `pgbench`, it should be configured with a custom script file that simulates these exact workloads, allowing for a meaningful comparison between UUIDv7 and its practical alternatives. |
| 25 | Is migration from UUIDv4 to UUIDv7 a risky and difficult process that will break my application? | No, the migration is often surprisingly low-risk and easy. Since UUIDv4 and UUIDv7 are both 128-bit UUIDs, they are indistinguishable to most applications and databases at the storage level. You can typically start generating UUIDv7 for new records while keeping existing UUIDv4 IDs. They are fully compatible and can coexist in the same column, so cascading key updates are unnecessary.<br><br>However, be aware of three caveats:<br><br>If your application extracts the timestamp from UUIDs, existing UUIDv4 values will not contain a valid timestamp.<br><br>You may need to remove any constraints that enforce a specific UUID version.<br><br>If you shard using the full UUID as a sharding key, switching to UUIDv7 can create hotspots. In that case, use only the 32 least significant bits (the random part) as the sharding key. |
| 26 | If I use UUIDv7, can I still generate IDs on the client-side without a round-trip to the database, unlike a BIGINT? | Yes. UUIDv7 generates the timestamp and random bits locally, so no central sequence or database call is needed. A BIGINT typically requires a database request for the next value. |
| 27 | How can I improve the monotonicity of UUIDv7 generated on remote clients for specific use cases like reliable cursor‑based pagination, if strict monotonicity is critical? | The monotonicity of UUIDv7 generated on remote clients can be significantly improved by using a positive timestamp offset equal to half the ping time (RTT/2). This offset compensates for the one‑way network delay, making the timestamps of client‑generated UUIDs align with the timestamps a server would have assigned upon reception. Server‑side generation ensures better monotonicity, though it may require additional client‑generated identifiers for integrity and feedback. |
| 28 | Is it necessary to synchronize the system clocks of UUIDv7 generators with each other? | No, UUIDv7 generators do not need to synchronize their clocks with each other directly. Instead, each generator independently synchronizes its system clock to a global time reference (typically UTC) using standard time synchronization technologies, such as NTP (Network Time Protocol), PTP (Precision Time Protocol), or GPS-based time receivers. This ensures that all generators produce monotonically increasing timestamps without requiring peer‑to‑peer synchronization. |
| 29 | When to choose UUIDv7 vs UUIDv4? | UUIDv7 is best for primary keys in tables that handle a lot of writes. Because it's time-ordered, it improves index performance and reduces page splits in databases like PostgreSQL or MySQL.<br><br>UUIDv4 should only be used when you need purely random identifiers with no sortable or time-based structure, for example, session tokens, API keys, or one-time use codes where ordering doesn't matter. |
| 30 | Does `Guid.CreateVersion7()` in .NET 9 produce a sortable UUIDv7 for SQL Server `uniqueidentifier` columns, and if not, what is the correct alternative? | No. `Guid.CreateVersion7()` stores the timestamp in little‑endian order, violating RFC 9562's big‑endian requirement, and SQL Server's `uniqueidentifier` type reorders bytes during comparison, both of which prevent lexicographical time‑sorting. To achieve correct chronological sorting, generate UUIDv7 using a third‑party library that follows the RFC (e.g., UUIDNext, Medo.Uuid7, Costasdev.Uuidv7) and store the 16‑byte array in a `binary(16)` column instead of `uniqueidentifier`. |
| 31 | Some UUIDv7 implementations allow generation from a specific timestamp, while others offer a timestamp offset. Considering they might seem interchangeable, which approach is preferable and why? | Choose timestamp offset, because generation from a specific timestamp is a safety anti-pattern. Unintentional passing of an arbitrary (fixed or decreasing) timestamp to uuidv7() fundamentally breaks monotonicity, the backbone of time-based sortable UUIDs. So the result cannot be considered a valid UUIDv7 implementation. |
| 32 | How do UUIDv7 implementations preserve monotonic order at very high generation frequencies? | There are three methods that can be combined. The first uses a randomly seeded counter that increments for each UUID generated within the same millisecond. The second uses an additional sub-millisecond fraction of the timestamp to create a finer grained clock. The third increments the timestamp itself under extreme generation rates. |
| 33 | Is it necessary to implement multiple UUID versions in DBMS and system libraries? | As a rule, implementing only UUIDv7 is sufficient.<br><br>Versions 1-5 have flaws and are not recommended for new systems (de facto deprecated).<br><br>UUIDv4 works well in those rare cases where ordering is not required. However, proper use of UUIDv7 can completely replace it.<br><br>UUIDv6 is unpopular, and in the vast majority of cases can be replaced by UUIDv7 without any drawbacks.<br><br>UUIDv8 is custom and therefore cannot be uniformly implemented in DBMS and system libraries. |
